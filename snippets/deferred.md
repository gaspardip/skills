## Deferred work tracking

When the user explicitly defers something ("skip that for now", "not now", "we'll handle that later", "we'll come back to that", "leave that for now", "punt on that", "defer", "come back to that", "park that", "backlog that"), append a JSONL entry to `~/.claude/projects/<project-key>/deferred.jsonl` with fields: `summary` (imperative, <80 chars), `context` (1-2 sentences), `project` (cwd), `timestamp` (ISO-8601), `branch` (current git branch), `sha` (short commit hash). Only track when the user is explicitly choosing to postpone a specific piece of work — not when rejecting a suggestion, discussing language keywords (e.g. Go's `defer`), or using these phrases conversationally. Good summary: "Add retry logic to webhook API client". Bad: "fix that thing". Briefly confirm: "Noted — deferred: \<summary\>". Do not interrupt flow. If the deferred file for the current project has 5+ unreviewed entries, mention it once at session start: "You have N deferred items — run /deferred to review."
